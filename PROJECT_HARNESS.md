# jub RN Project Harness

이 문서는 `jub` 프로젝트에서 내가 담당하는 **React Native 클라이언트 구현 범위만** 고정하기 위한 하네스다. 목적은 제품 전체를 다시 정의하는 것이 아니라, RN에서 무엇을 구현하고 무엇을 구현하지 않을지 선명하게 정하는 데 있다.

## 1. 프로젝트의 범위

### 1.1 이 프로젝트가 다루는 범위

- 대상 사용자: 소비 통제가 어렵지만 죄책감 기반 절약 앱은 싫어하는 20대 사회초년생
- 플랫폼: 모바일 앱 우선. iOS/Android 동시 대응을 전제로 한 단일 제품 경험
- 제품 역할: 돈을 덜 쓰게 강요하는 앱이 아니라, 어떤 소비가 더 행복으로 이어지는지 학습하고 보여주는 앱
- RN 담당 범위: 화면, 내비게이션, 상태 관리, 폼 입력, UI 피드백, API 연동 계층, 이벤트 트래킹 포인트 정의
- 핵심 사용 시점: 결제 전 챗봇 상담, 주 1회 회고. 입력 채널은 이 2개만 유지
- RN 산출물: 사용자가 실제로 조작할 수 있는 모바일 화면과 그 화면을 잇는 상태 흐름

### 1.2 RN 구현에 반드시 포함되는 범위

- 온보딩 화면 플로우
- 최근 거래/회고 후보를 보여주는 초기 라벨링 UI
- 결제 전 챗봇 상담 화면
- `살게요 / 다시 생각 / 안 살게요` 종료 액션 UI
- 챗봇 상담 요약 노출 UI
- 일요일 회고 화면과 점수 입력 UI
- 행복 소비 아카이브 화면
- 안 쓴 돈 카운터 화면
- 첫 5회 풀기능, 이후 일부 제한 상태를 표현하는 프론트 UI
- 로딩, 에러, 빈 상태, 데이터 부족 상태 UI

### 1.3 RN 구현에서 제외되는 범위

- 실제 오픈뱅킹 백엔드 연동 구현
- 서버 데이터베이스 설계와 마이그레이션
- LLM 프롬프트 엔지니어링과 모델 운영
- 추천/분류/큐레이션 로직의 서버 측 계산 구현
- 결제 여부 판별용 금융 데이터 파이프라인
- 관리자 콘솔, 내부 운영 툴
- 과금 결제 서버 구현

### 1.4 RN이 전제로 두는 외부 시스템

- 거래 내역을 내려주는 백엔드 또는 mock provider
- 챗봇 응답과 요약을 내려주는 AI API
- 회고 후보를 내려주는 큐레이션 API
- 사용자 인증 세션
- 이벤트 수집 엔드포인트

### 1.5 RN 산출물 범위

- 사용자 앱 화면 구조
- 컴포넌트 구조
- 프론트 상태 모델 기준
- 화면별 API contract 요구사항
- 제품 카피 톤 반영 규칙
- 이벤트 트래킹 삽입 지점

## 2. 절대하면 안되는 것

- 절약 앱처럼 행동하지 않는다. "사지 마세요", "후회할 거예요" 같은 경고형 UX 금지
- 결제 직후 점수 입력을 되살리지 않는다
- 입력 채널을 2개 이상으로 늘리지 않는다. MVP에서는 챗봇 상담과 주간 회고만 허용
- 챗봇을 친구처럼 들뜨게 만들지 않는다. 톤은 담담한 상담사여야 한다
- 사용자의 결정을 대신 내려주지 않는다. AI는 데이터와 경향을 제시하고 결정은 사용자가 한다
- 필수 소비를 회고의 주력 대상으로 삼지 않는다
- 가계부 기능을 확장해서 범용 자산관리 앱으로 변질시키지 않는다
- 콜드 스타트 문제를 무시한 채 챗봇을 바로 열지 않는다. 최소 5개 라벨 이후 활성화가 원칙이다
- "행복" 대신 "후회 방지"를 전면 메시지로 쓰지 않는다
- PMF 검증 전 고급 추천, 소셜 기능, 게임화, 리워드 체계를 붙이지 않는다
- RN 앱 안에 비밀키를 넣지 않는다
- RN에서 추천 로직과 점수 계산 로직을 임의로 복제하지 않는다. 계산은 서버 응답을 따르고, RN은 표현과 입력 흐름에 집중한다
- 백엔드가 정해지지 않았다는 이유로 화면 요구사항을 흐리게 만들지 않는다. 필요한 contract는 명시적으로 요구한다

## 3. 사용할 기술스택

아래는 `jub`의 RN 구현 기준 권장 스택이다. 모바일 앱 개발 속도, 상태 단순성, API 연동 용이성을 우선한다.

| 영역 | 스택 | 선택 이유 |
| --- | --- | --- |
| 모바일 앱 | React Native + Expo + TypeScript | 하나의 코드베이스로 iOS/Android를 빠르게 검증하기 좋음 |
| 라우팅 | Expo Router | 화면 단위 구조화와 모바일 플로우 관리에 적합 |
| 상태 관리 | Zustand | 챗봇 상태, 회고 큐, 세션성 UI 상태를 다루기 가볍고 명확함 |
| 서버 상태 | TanStack Query | 거래, 회고 큐, 챗봇 응답 등 비동기 API 상태 관리에 적합 |
| 폼 처리 | React Hook Form | 회고 점수 입력과 챗봇 입력 폼을 단순하게 관리 가능 |
| API 클라이언트 | fetch wrapper 또는 axios | 백엔드 contract를 분리하고 에러 처리 공통화 가능 |
| 푸시 알림 | Expo Notifications | 일요일 회고 리마인드와 재진입 유도 UI 연결에 충분 |
| 분석 | PostHog 또는 Segment SDK | 챗봇 시작/종료, 회고 완료 등 제품 이벤트 기록 필요 |
| 에러 모니터링 | Sentry | 모바일 앱 장애 및 API 실패 추적에 필요 |
| 디자인 시스템 | RN 공용 토큰 + StyleSheet 또는 Tamagui 중 하나로 통일 | 베이지/세이지 톤과 반복 UI를 일관되게 유지하기 위함 |

### 3.1 스택 결정 원칙

- RN 안에 비즈니스 계산 로직을 과도하게 넣지 않는다
- 서버 상태와 로컬 UI 상태를 분리한다
- API schema가 바뀌더라도 화면이 전부 무너지지 않게 adapter 계층을 둔다
- 초기에는 mock 응답과 실제 응답을 같은 interface로 다룬다

## 4. 프로젝트의 정확한 목표

### 4.1 RN 구현 목표

`jub` RN 앱의 목표는 PRD의 핵심 경험을 모바일에서 자연스럽게 재현하는 것이다. 즉, 사용자가 결제 전 상담과 주간 회고를 부담 없이 오가면서 자기 소비 패턴을 읽을 수 있게 만드는 것이 목표다.

### 4.2 RN MVP 목표

- 사용자가 결제 전에 챗봇에 소비 고민을 입력할 수 있다
- 앱이 AI 응답을 담담한 상담사 톤으로 안정적으로 표시한다
- 사용자는 `살게요 / 다시 생각 / 안 살게요` 중 하나로 상담을 끝낼 수 있다
- 상담 종료 후 요약 결과를 다시 확인할 수 있다
- 일요일 회고에서 제시된 소비 몇 개를 빠르게 평가할 수 있다
- 앱은 누적 데이터를 바탕으로 "내가 잘 쓴 돈"과 "막은 돈"을 보여줄 수 있다

### 4.3 RN 성공 판정 기준

- 사용자가 주요 플로우를 막힘 없이 완료하는가
- 챗봇과 회고 화면의 정보 밀도가 높아도 피로하지 않은가
- 앱의 톤이 죄책감이 아니라 발견과 안도감을 주는가
- mock/real API 어느 쪽에서도 화면 상태가 일관되게 유지되는가

### 4.4 핵심 지표

- RN에서 반드시 트래킹해야 할 이벤트:
  - 챗봇 상담 시작
  - 챗봇 상담 응답 수신 성공/실패
  - 챗봇 종료값 선택
  - 회고 진입
  - 회고 완료
  - 아카이브 조회
  - 안 쓴 돈 카운터 조회

## 5. 프로젝트의 정확한 기능들

### 5.1 기능 1: 온보딩과 콜드 스타트 해소 화면

- 사용자는 먼저 오픈뱅킹을 연결한다
- 앱은 최근 거래를 가져오고, 과거 소비 중 일부를 회고 후보로 제시한다
- 사용자는 "만족스러웠던 소비"를 최소 5개 라벨링한다
- 5개 이상 라벨이 쌓이면 챗봇 기능이 열린다
- 첫 인사이트 카드를 통해 "당신은 어디에 쓸 때 만족도가 높다"는 메시지를 보여준다
- RN은 라벨 진행도, 비활성/활성 상태, 데이터 부족 상태를 명확히 표현해야 한다

### 5.2 기능 2: 결제 전 챗봇 상담 화면

- 사용자는 자유 텍스트로 고민 중인 소비를 입력한다
- 시스템은 서버 응답으로 받은 상품/금액/카테고리/소비 유형 정보를 표시할 수 있어야 한다
- AI 응답은 과거 패턴과 일반 통계를 조합한 정보형 응답으로 노출되어야 한다
- 응답은 반드시 데이터 중심 문장이어야 하며, 감정 과잉 공감형 문장을 쓰지 않는다
- 상담은 반드시 다음 3개 버튼 중 하나로 종료된다
  - `살게요`
  - `다시 생각`
  - `안 살게요`
- 종료 후 RN은 저장된 요약본을 다시 조회하거나 즉시 보여줄 수 있어야 한다
- 요약 카드에는 최소한 다음 정보가 보여야 한다
  - 어떤 소비였는지
  - 사용자가 왜 끌렸는지
  - AI가 어떤 비교 기준을 제시했는지
  - 최종 결정이 무엇이었는지
- 로딩 중, 응답 실패, 데이터 부족 상태가 별도로 설계되어야 한다

### 5.3 기능 3: 거래 분류 결과 표현

- 모든 거래는 아래 셋 중 하나의 결과를 RN에서 표시할 수 있어야 한다
  - 즉시 소비형
  - 지속 소비형
  - 필수 소비형
- 필수 소비형은 회고 추천 대상에서 제외된다는 사실이 UI에서도 자연스럽게 드러나야 한다
- 즉시 소비형과 지속 소비형은 서로 다른 설명 문구를 가져야 한다

### 5.4 기능 4: 일요일 회고 화면

- 회고는 매주 일요일 저녁에만 명확하게 호출한다
- 회고 대상 개수는 사용자가 정하지 않고 시스템이 정한다
- 회고 선정 로직은 서버가 아래 4가지를 반영해 내려준다고 가정한다
  - 만족 강화
  - 공백 메우기
  - AI가 헷갈리는 소비
  - 다양성 확보
- 각 항목은 1-5 점수 입력이 필수다
- 메모 한 줄은 선택이다
- 챗봇에서 `살게요`한 소비는 회고에서 우선적으로 다시 보여준다
- 이때 챗봇 요약과 함께 "예상 vs 실제 만족도"를 비교할 수 있어야 한다
- RN은 여러 항목을 빠르게 넘기며 입력할 수 있는 흐름을 보장해야 한다

### 5.5 기능 5: 행복 소비 아카이브 화면

- 만족도가 높게 누적된 소비를 카드 형태로 보여준다
- 이 화면의 목적은 사용자가 "나는 어디에 쓸 때 행복한가"를 빠르게 복기하게 하는 것이다
- 단순 거래 리스트가 아니라, 만족 패턴이 보이도록 정리해야 한다
- 사용자가 평소 앱을 여는 주요 이유 중 하나여야 한다
- RN은 카테고리, 만족도, 한 줄 회고, 관련 챗봇 요약을 시각적으로 묶어서 보여줄 수 있어야 한다

### 5.6 기능 6: 안 쓴 돈 카운터 화면

- 챗봇에서 `안 살게요`로 종료한 금액을 누적한다
- 단순 절약 금액이 아니라 "막은 돈"으로 해석한다
- 가능하면 "이 돈이면 만족도 상위 영역에 이렇게 쓸 수 있었다"는 대안 메시지를 붙인다
- 사용자가 평소 앱을 여는 두 번째 이유가 되어야 한다
- RN은 누적 금액, 최근 막은 소비, 대안 메시지를 한 화면에서 이해되게 구성해야 한다

### 5.7 기능 7: 역방향 freemium UI

- 첫 5회 챗봇 상담은 풀기능으로 제공한다
- 6회 이상부터는 일부 인사이트와 차트를 제한한다
- 유료 전환 시 다시 풀기능으로 복귀한다
- 이 구조는 확정 비즈니스 모델이 아니라 실험 대상이다
- RN은 남은 풀기능 횟수, 제한된 상태, 업그레이드 유도 UI를 표현할 수 있어야 한다

### 5.8 기능 8: 이벤트 트래킹과 API 상태 처리

- 최소 트래킹 이벤트
  - 챗봇 시작
  - 챗봇 종료값
  - 챗봇 요약 노출
  - 회고 노출
  - 회고 완료
  - 아카이브 조회
  - 안 쓴 돈 카운터 조회
- RN은 성공/실패/재시도 상태를 화면에서 구분해야 한다
- 학습 데이터 계산 자체는 서버 책임이며, RN은 필요한 이벤트를 정확히 발생시킨다

## 6. 한 줄 운영 원칙

`jub` RN 앱은 사용자를 말리는 앱처럼 보이면 안 되고, 사용자가 자기 행복 패턴을 더 정확히 읽게 만드는 인터페이스여야 한다.
